# ArchGuard Methodology — Source of Truth

The single source of truth for the Architectural Guard Methodology. This is the immutable constitution for AI-native development. All skills (`archguard-discover`, `archguard-clarify`, `archguard-plan`, `archguard-implement`, `archguard-analyze`) derive their behavior from this document.

---

## Part 1: The 6 Failure Modes (The Axioms)

These are universal. They apply to every project, in every domain. They define *why* agent shortcuts are dangerous and *what tests cannot see*.

| # | Failure Mode | The Agent Shortcut | The Invariant | The CS Principle |
|:---|:---|:---|:---|:---|
| **FM1** | **Trusts Input** | Passes raw external data into internal state without validation. | **I.1 Data Boundary** | Complete Mediation (Saltzer & Schroeder, 1975) |
| **FM2** | **Inverts Dependencies** | Imports concrete infrastructure into abstract business logic. | **I.2 Dependency Direction** | Dependency Inversion (Martin, 1996) / Information Hiding (Parnas, 1972) |
| **FM3** | **Leaks State** | Dumps raw internal state (PII, secrets) into logs, responses, or telemetry. | **I.3 Leakage Gate** | Lattice Model of Information Flow (Denning, 1976) |
| **FM4** | **Ignores Failure** | Makes external calls without timeouts; writes empty catch blocks; skips circuit breakers. | **I.4 Resilience** | Fail-Fast (Gray, 1985) / Circuit Breaker (Nygard, 2007) |
| **FM5** | **Skips Cleanup** | Opens connections/streams without guaranteed teardown. | **I.5 Resource Lifecycle** | RAII (Stroustrup) / Deterministic Destruction |
| **FM6** | **Writes Partial State** | Executes multiple mutations sequentially without atomicity. | **I.6 State Mutation** | ACID (Gray & Reuter, 1992) / Linearizability |

### Formal Completeness Proof

Any statically analyzable program is defined by its **Control Flow**, **Data Flow**, **Dependency Graph**, and **Resource Lifecycle**. Architectural violations always manifest as a failure in one of these dimensions:

```
State Machine = f( Input, Current State, Resources ) → Output + New State
```

| FSM Component | Requirement | Failure Mode |
|:---|:---|:---|
| **Input** | Validated before entering state machine | FM1 (Trusts Input) |
| **Internal Logic** | Correct dependency references | FM2 (Inverts Dependencies) |
| **Output** | No exposure of internal state | FM3 (Leaks State) |
| **Transition** | Graceful error handling + timeouts | FM4 (Ignores Failure) |
| **Resources** | Released after transition completes | FM5 (Skips Cleanup) |
| **State Persistence** | Atomic updates | FM6 (Writes Partial State) |

Any 7th failure mode can be classified into one of these 6. Example:
- "SQL Injection" → FM1 (untrusted data in query)
- "Hardcoded secrets" → FM3 (leaked to source code output)
- "Infinite loops" → FM4 (no boundary on transition)
- "Missing idempotency" → FM6 (no atomicity across retries)

## Part 2: The 4 Structural Maps

To find where the FMs will manifest, scan the codebase for these 4 structural maps. These are generated by `archguard-discover` via pattern matching — no domain knowledge required.

### Map A: Trust Boundaries (data enters the system)

| Category | Patterns to find | FM |
|:---|:---|:---|
| HTTP request objects | `req.body`, `req.params`, `request.json()`, `ctx.request.body` | FM1 |
| WebSocket / realtime events | `.on('message')`, `.onmessage`, `.subscribe(` | FM1 |
| CLI / process arguments | `process.argv`, `process.env` reads | FM1 |
| File system reads | `fs.readFile`, `fs.createReadStream` | FM1 |
| Job queue consumers | Worker `process()` callbacks | FM1 |
| Browser / platform events | `addEventListener(`, `postMessage` handlers | FM1 |

### Map B: Dependency Graph (import relationships)

1. Identify layer boundaries from directory structure
2. Classify each directory: Domain/Core, Infrastructure/Adapters, Entrypoints/Transport, Features/Modules, Apps/UI, Scripts
3. Determine allowed dependency directions (inner → outer is banned)
4. Flag all cross-boundary imports

| Violation | FM |
|:---|:---|
| Domain layer importing from Infrastructure | FM2 |
| Server-side code importing browser-only libraries | FM2 |
| Entrypoints importing from other entrypoints | FM2 |

### Map C: Output Surfaces (data leaves the system)

| Category | Patterns to find | FM |
|:---|:---|:---|
| HTTP responses | `.json()`, `.send()`, `return` in route handlers | FM3 |
| Logging calls | `.info(`, `.error(`, `.warn(`, console methods | FM3 |
| Telemetry / tracing | `span.setAttribute(`, `span.addEvent(`, `setTag(` | FM3 |
| WebSocket outbound | `.send(`, `.emit(`, `.broadcast(` | FM3 |
| Error propagation | Error constructors with metadata, `throw new` with details | FM3 |

### Map D: Resources + Mutations (connections opened, state written)

| Category | Patterns to find | FM |
|:---|:---|:---|
| Database mutations | For EACH DB driver found: `.insert(`, `.update(`, `.delete(`, `.execute(`, `.run(` | FM6 |
| Stream/file handles | `createWriteStream`, `createReadStream` | FM5 |
| Crypto operations | `createCipheriv`, `generateKeyPair` | FM5 |
| Realtime connections | For EACH realtime library: connect/open/join methods | FM5 |
| Process termination | `process.exit(`, runtime-specific exit calls | FM5 |
| External network calls | `fetch(`, framework-specific HTTP clients | FM4 |

**FM6 detection (multi-mutation):** Flag any function body with 3+ consecutive write calls without a transaction wrapper.

## Part 3: Tech Context — The 5 Escape Hatch Questions

For every dependency in `package.json`, apply these 5 universal questions. The questions are fixed. The answers come from the dependency's actual API surface.

| # | Question | If "Yes" | FM |
|:---|:---|:---|:---|
| **Q1** | **Raw escape hatch?** Can raw/untrusted strings reach an engine/interpreter? | Ban raw string interpolation; force parameterized API | FM1 |
| **Q2** | **Unbounded default?** Can it be created without explicit limits? | Ban instantiation without maxSteps/maxTokens/maxRetries/maxConnections | FM4 |
| **Q3** | **Trust boundary?** Is output treated as safe without sanitization? | Ban passing output directly to storage/responses without sanitization | FM1 / FM3 |
| **Q4** | **Client/server split?** Are there platform-specific variants? | Ban platform-specific imports in wrong context | FM2 |
| **Q5** | **Missing timeout?** Can it be called without a deadline/abort? | Ban calls without timeout/signal/AbortController | FM4 |

Dependencies that pass all 5 questions need no bans. That is fine — most dependencies are not risks.

## Part 4: The 13 Invariant Bans (Universal, Non-Negotiable)

These are the concrete syntactic bans that enforce the 6 Failure Modes. They are auto-generated by `archguard-discover` and included by `archguard-plan`. No user confirmation is needed — they are axioms.

| # | Ban | FM | ESLint Rule / Selector | Activation |
|:---|:---|:---|:---|:---|
| 1 | Raw `JSON.parse()` | FM1 | `CallExpression[callee.object.name='JSON'][callee.property.name='parse']` | Always |
| 2 | `any` type + `as any` | FM4 | `@typescript-eslint/no-explicit-any: error` | Always |
| 3 | `@ts-ignore` / `@ts-nocheck` | FM4 | `@typescript-eslint/ban-ts-comment: error` | Always |
| 4 | `console.log` (and info/warn/debug) | FM3 | `CallExpression[callee.object.name='console'][callee.property.name=/^(log\|info\|warn\|debug)$/]` | Always |
| 5 | `fetch()` without `AbortSignal` | FM4 | `CallExpression[callee.property.name='fetch']:not([arguments.1.properties[name='signal']])` | If `fetch()` used |
| 6 | Empty `catch` blocks | FM4 | `CatchClause[body.body.length=0]` | Always |
| 7 | `process.exit()` / `Bun.exit()` | FM5 | `CallExpression[callee.object.name='process'][callee.property.name='exit']` | Node.js / Bun |
| 8 | `while(true)` / `for(;;)` + `setInterval` | FM4/FM5 | `WhileStatement[test.value=true]`, `ForStatement[init=null][test=null][update=null]`, `CallExpression[callee.name='setInterval']` | If found |
| 9 | `process.env` outside config | FM1 | `MemberExpression[object.object.name='process'][object.property.name='env']` (excl. config dir) | Node.js |
| 10 | `Date.now()` / `new Date()` in business logic | FM4 | `NewExpression[callee.name='Date']`, `CallExpression[callee.object.name='Date'][callee.property.name='now']` | Deferred to `clarify` |
| 11 | Floating promises (un-awaited) | FM5/FM4 | `@typescript-eslint/no-floating-promises: error` | Always |
| 12 | `export *` (barrel exports) | FM3 | `ExportAllDeclaration` | Always |
| 13 | Mutable module-level state (`let`/`var` at module scope) | FM6 | `VariableDeclaration[declarations.kind!='const']:not(:has(ancestor::BlockStatement))` | Always |

### Per-primitive external client bans (from Tech Context Q1-Q5)

These are NOT in the invariant table because the specific selectors depend on which packages are installed. `discover` generates them via the 5 Questions. Examples (not exhaustive):

- Neo4j: `session.run()` with template expressions → FM1
- Prisma: `$queryRaw`, `$executeRaw` → FM1
- Supabase: `.rpc()` bypassing RLS → FM1
- Mastra: `new Agent()` without `maxSteps` → FM4
- Vercel AI SDK: `generateText()` returned raw → FM1
- LiveKit: `livekit-client` in server paths → FM2

## Part 5: The Whitelist Rule

**Every ban must have exactly ONE whitelist file** where the banned primitive is allowed.

| Primitive | Whitelist File (example paths) | Safety Wrapper |
|:---|:---|:---|
| `fetch()` | `<adapter-layer>/http/safe-fetch.ts` | Timeout (AbortSignal) + Zod validation |
| `JSON.parse` | `<core-layer>/safe-parse.ts` | JSON.parse inside Zod.safeParse |
| `process.env` | `<config-layer>/env-schema.ts` | Centralized env parsing + validation |
| `console.log` | `<project-logger-file>` | Structured logger with trace IDs |
| `process.exit` | `<core-layer>/shutdown.ts` | Graceful SIGTERM/SIGINT handler |
| `Date.now()` / `new Date()` | `<core-layer>/time-service.ts` | Injectable time abstraction |
| DB driver queries | `<adapter-layer>/<db>/parameterized-query.ts` | Parameterized queries only |

The whitelist stub enforces what the ban requires — timeout + signal + validation. Implementation details (retry, backoff, idempotency keys) are operational decisions per-endpoint, not part of the architectural contract.

## Part 6: Separation of Concerns

This methodology covers **structural invariants** — agent shortcuts that pass tests but break production. Related concerns live in separate documents:

| Concern | Document | Tool |
|:---|:---|:---|
| Structural invariants (FM1-FM6) | `archguard-methodology.md` (this file) | ESLint + ArchUnit |
| Code hygiene (complexity, nesting, dead code) | `archguard-hygiene.md` | ESLint + Knip |
| Pipeline (5 skills + data flow) | `archguard-pipeline.md` | archguard-* skills |

All three flow into one `eslint.config.cjs`. The separation is for documentation clarity, not deployment.
