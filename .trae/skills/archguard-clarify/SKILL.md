# ArchGuard Clarify

Phase 3 of the architectural guard generator. Reads the discover output and walks the user through the 6 Failure Modes, asking one question per mode based on the actual code patterns found. The 6 FMs are universal. The evidence comes from discover. Only the user's answers are domain-specific.

## Pipeline Position

```
archguard-discover → archguard-clarify → archguard-plan → archguard-implement
                              ↑ You are here
```

## Prerequisites

- `.archguard/discover.md` must exist (generated by archguard-discover).
- `.archguard/maps/*.json` should exist if available — use them for precise counts and file lists. Fall back to discover.md if JSON maps are missing (first run before discover was updated).
- The invariant-derived bans from discover are **not discussed** — they are axioms. Clarify only covers the 6 FMs and domain context.

## The 6 Failure Modes (Fixed, Universal)

These never change. They are the axioms. For each, ask the user one question. The question must reference specific evidence from discover — never generic hypotheticals.

### FM1: Trusts Input (Data Boundary)

The agent shortcut: passes raw external data directly into internal state without validation.

```
Read from discover:
  - Map A count and categories
  - Tech Context Q1 + Q3 entries (raw escape hatches, trust boundary libraries)

Ask the user:

"I found [N] places where data enters the system from outside.
 Categories found: [list from Map A]

 Which of these entry points are exposed to EXTERNAL / untrusted users?"

Options:
  (1) All entry points are external — ban raw input processing everywhere
  (2) Some are internal/trusted — we need per-location exceptions
  (3) Not sure — apply ban to all, we'll whitelist trusted locations later
```

**Decision recorded as:** FM1 scope + list of external boundary locations.

### Special: Date.now / new Date (Invariant Ban #10)

This ban is deferred from discover because TimeService abstraction is architectural, not invariant.

```
Ask the user:

"I found [N] uses of Date.now() or new Date() in business logic files
 (excluded: __tests__, time-service implementations).

 Should we require a TimeService abstraction for testability?"

Options:
  (1) Yes — ban direct Date.now()/new Date() in non-test, non-time-service files.
      Whitelist: <time-service-file>
  (2) No — this project doesn't need time abstraction. Skip this ban.
```

**Decision recorded as:** Ban #10 active/inactive + whitelist path if active.

### FM2: Inverts Dependencies (Dependency Direction)

The agent shortcut: imports concrete infrastructure into abstract business logic.

```
Read from discover:
  - Map B layer structure and FM2 candidates
  - Tech Context Q4 entries (platform split violations)

Ask the user:

"I identified these layer boundaries: [list from Map B].
 I found [N] imports that cross from inner layers to outer layers:
 [List top FM2 candidates from Map B]

 Are these cross-layer imports accidental (agent mistakes)
 or intentional (this project allows them)?"

Options:
  (1) All accidental — enforce strict dependency direction
  (2) Some intentional — mark specific imports as exceptions
  (3) This project doesn't enforce layer boundaries — skip FM2 entirely
```

**Decision recorded as:** FM2 scope + list of exceptions (if option 2).

### FM3: Leaks State (Output Sanitization)

The agent shortcut: dumps raw internal state (PII, secrets, internal IDs) into logs, responses, or telemetry.

```
Read from discover:
  - Map C categories and counts
  - Scan discover output for PII-like field names mentioned in Map C matches

Ask the user:

"I found [N] output surfaces where data leaves the system.
 I also found references to potentially sensitive field names:
 [List PII-like identifiers found in Map C context snippets:
  e.g., email, password, token, phone, ssn, secret, apiKey]

 What data must NEVER leave the system in raw form?"

Options:
  (1) Standard PII set: email, phone, password, token, secret, api_key — ban in all outputs
  (2) Custom fields: [user specifies additional sensitive fields]
  (3) Not applicable — no sensitive data in this system, skip FM3
```

**Decision recorded as:** FM3 sensitive field list + output surfaces to protect.

### FM4: Ignores Failure (Resilience)

The agent shortcut: makes external calls without timeouts, writes empty catch blocks, skips circuit breakers.

```
Read from discover:
  - Map D external call counts (network, DB, AI)
  - Tech Context Q2 + Q5 entries (unbounded defaults, missing timeouts)
  - Count of empty catch blocks (if detectable from file scan)

Ask the user:

"I found [N] external calls (HTTP requests, DB queries, AI/LLM calls)
 and [M] empty or untyped catch blocks.

 Which failure modes are critical for this system?"

Options:
  (1) All — enforce timeouts on every external call, non-empty catch with : unknown type,
      circuit breaker on all adapter calls
  (2) Only network calls — trust DB drivers and framework timeouts
  (3) Minimal — just enforce no empty catch blocks
```

**Decision recorded as:** FM4 scope + which layers need circuit breaker.

### FM5: Skips Cleanup (Resource Lifecycle)

The agent shortcut: opens connections, streams, or crypto operations without guaranteed teardown.

```
Read from discover:
  - Map D resource categories (streams, realtime connections, crypto)
  - process.exit / ungraceful shutdown patterns

Ask the user:

"I found [N] resource allocations ([list categories from Map D]).
 I also found [Y] ungraceful shutdown calls (process.exit etc).

 Does this project have long-lived resources that must survive crashes?"

Options:
  (1) Yes — we use persistent connections (WebSocket, WebRTC, message queues)
      → enforce try/finally cleanup + graceful shutdown handlers
  (2) Only connection pools — the pool manager handles lifecycle, but ban process.exit
  (3) No long-lived resources — just ban process.exit, skip resource lifecycle checks
```

**Decision recorded as:** FM5 scope + which resource types need lifecycle enforcement.

### FM6: Writes Partial State (Transaction Integrity)

The agent shortcut: executes multiple database mutations sequentially without atomicity.

```
Read from discover:
  - Map D FM6 candidates (3+ consecutive mutations in same function body)

Ask the user:

"I found [N] functions with 3+ sequential database mutations:
 [List top FM6 candidates from Map D]

 Do these mutations need to be atomic (all succeed or all roll back)?"

Options:
  (1) Yes — enforce transaction/atomic wrapper on all multi-mutation functions
  (2) Not applicable — mutations in this project are independent by design
  (3) Our ORM/database handles atomicity at the driver level — skip FM6
```

**Decision recorded as:** FM6 scope + list of functions requiring atomicity.

## Constraints

- **Ask all 6 questions.** Even if a FM seems irrelevant to you, let the user decide.
- **Reference discover evidence.** Every question must cite specific Map results or Tech Context entries. No generic "somewhere in your codebase" questions.
- **Use the project's own terminology.** If Map B calls layers "domain/" and "infra/", use those names, not "core/" and "adapters/".
- **Do NOT modify config files.** This skill only collects domain context.
- If discover output is missing, abort and tell the user to run archguard-discover first.
