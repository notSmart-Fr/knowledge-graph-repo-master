# ArchGuard Clarify

Phase 3 of the architectural guard generator. Reads the discover output and walks the user through the 6 Failure Modes, asking one question per mode based on the actual code patterns found.

## Pipeline Position

```
archguard-discover → archguard-clarify → archguard-plan → archguard-implement
                              ↑ You are here
```

## Prerequisites

- `.archguard/discover.md` must exist (generated by archguard-discover).
- The user should have context about their project's domain (CRM, CLI, game, etc.).

## The 6 Failure Modes (Fixed, Universal)

These never change. They are the axioms. For each, ask the user one question based on what discover found:

### FM1: Trusts Input (Data Boundary)

```
Question: "I found [N] places where untrusted data enters the system.
          Which of these are exposed to EXTERNAL users (vs. internal/trusted)?

          [List top 5 files with boundary patterns]

Options:  (1) All are external — ban raw input everywhere
          (2) Some are internal — we need per-file exceptions
          (3) Not sure — apply ban to all, we'll whitelist later"
```

**Rule generated:** Ban raw `JSON.parse`, `req.body` without Zod, `ws.on('message')` without schema on all external boundaries.

### FM2: Inverts Dependencies (Dependency Direction)

```
Question: "I found [N] imports that cross layer boundaries
          (e.g., core/ importing from adapters/ or features/).

          [List FM2 candidates]

          Are these accidental (agent mistake) or intentional?

Options:  (1) All accidental — ban all cross-layer imports
          (2) Some intentional — mark as exceptions
          (3) This project doesn't enforce layer boundaries — skip FM2"
```

**Rule generated:** ArchUnit rules for dependency direction. ESLint `import/no-restricted-paths` if applicable.

### FM3: Leaks State (Output Sanitization)

```
Question: "I found PII-named variables and output surfaces.
          [List PII variable names found: email, password, token, etc.]
          [List output surfaces found: logger, res.json, spans]

          What data must NEVER leave the system unredacted?

Options:  (1) PII (email, phone, password, token) — standard set
          (2) Custom: [user specifies additional fields]
          (3) Not applicable — skip FM3"
```

**Rule generated:** PII-in-logs ban, PII-in-telemetry ban, PII-in-response ban.

### FM4: Ignores Failure (Resilience)

```
Question: "I found external calls (fetch, DB queries, AI APIs) and
          error handling patterns.
          [List external call count: N fetch(), M session.run()]
          [List empty catch blocks: P found]

          What failure modes are critical?

Options:  (1) All — enforce timeout + non-empty catch + circuit breaker
          (2) Only network calls — DB calls have built-in timeouts
          (3) Minimal — just enforce no empty catch blocks"
```

**Rule generated:** fetch-signal ban, empty-catch ban, circuit-breaker check where applicable.

### FM5: Skips Cleanup (Resource Lifecycle)

```
Question: "I found resource allocations (streams, connections, crypto)
          [List: createWriteStream, new Room, createCipheriv]

          Do you have long-lived connections (WebSocket, WebRTC, DB pools)?

Options:  (1) Yes, WebSocket/WebRTC — enforce try/finally + keepalive
          (2) Only DB connections — trust pool manager
          (3) No long-lived resources — skip FM5"
```

**Rule generated:** process.exit ban, floating-promise ban, keepalive check if applicable.

### FM6: Writes Partial State (Transaction Integrity)

```
Question: "I found [N] places where multiple mutations happen
          sequentially in the same function.

          [List files with 3+ consecutive .update/.insert/.delete calls]

          Do these require atomicity (all succeed or all roll back)?

Options:  (1) Yes — enforce transaction wrapper
          (2) Not applicable — mutations are independent
          (3) We use an ORM that handles this — skip FM6"
```

**Rule generated:** Multi-mutation without transaction → ban.

## Output

Write to `.archguard/clarify.md`:

```markdown
# ArchGuard Clarify Output

## User Decisions

### FM1: Trusts Input
- Decision: [1/2/3]
- External boundaries: [list of files/patterns]
- Whitelist files: [if any]

### FM2: Inverts Dependencies
- Decision: [1/2/3]
- Cross-layer imports to ban: [...]
- Exceptions: [...]

### FM3: Leaks State
- Decision: [1/2/3]
- Sensitive fields: [list]
- Output surfaces to protect: [...]

### FM4: Ignores Failure
- Decision: [1/2/3]
- Require timeouts on: [...]
- Circuit breaker scope: [...]

### FM5: Skips Cleanup
- Decision: [1/2/3]
- Long-lived resources: [...]

### FM6: Writes Partial State
- Decision: [1/2/3]
- Functions requiring atomicity: [...]
```

## Constraints

- **Ask all 6 questions**, even if some seem irrelevant. The user's answer determines if the rule is skipped.
- **Show the evidence.** For every question, list specific files/lines from discover so the user isn't guessing.
- **Do NOT modify config files.** This skill only collects domain context.
- If discover output is missing, abort and tell the user to run archguard-discover first.
