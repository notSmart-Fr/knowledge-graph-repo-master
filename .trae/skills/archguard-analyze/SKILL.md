# ArchGuard Analyze

Incremental analysis skill. Instead of re-running full discover every time a package or file changes, analyze diffs against the existing JSON maps and updates only what changed.

## Pipeline Position

```
archguard-discover → archguard-analyze → archguard-plan → archguard-implement
         ↓                    ↑
   (first run)      (incremental updates)
```

Run analyze after discover has created `.archguard/maps/*.json`. Analyze reports what's new, what's removed, and whether re-running clarify, plan, or implement is needed.

## Prerequisites

- `.archguard/maps/*.json` must exist (generated by archguard-discover).
- `.archguard/tech-context.json` must exist.
- `.archguard/discover.md` must exist (will be appended to).

## Input Modes

Analyze can be invoked with different scopes:

```
--pkg <name>       A new or updated dependency was added to package.json
--file <path>      A new file was added or an existing file was modified
--dir <path>       A new directory/module was added
--changed          Scan all files changed since last map generation (git diff)
--all              Full re-scan (delegates to discover — exit and tell user to run discover instead)
```

## Procedure

### Step 1: Load existing maps

Read all four JSON map files and tech-context.json into memory. These are the baseline.

### Step 2: Scan the target

#### If `--pkg <name>`:

1. Read `package.json` to find the package's actual installed version
2. Run the **5 Escape Hatch Questions** on this single package
3. Output: which Qs triggered, what bans are needed
4. **Search the codebase** for imports of this package — for each import found:
   - If the package is a DB driver → add to resources-mutations.json as DB query/mutation
   - If the package is an HTTP framework → add to trust-boundaries.json and output-surfaces.json
   - If the package is a realtime library → add to trust-boundaries.json and output-surfaces.json and resources-mutations.json
   - If Q4 triggered (platform split) → check if any import is in the wrong platform context

5. Append results to tech-context.json and affected map files

#### If `--file <path>`:

1. Read the file
2. Scan for patterns from all 4 map categories:
   - Map A: Does it contain trust boundary patterns?
   - Map C: Does it contain output surface patterns?
   - Map D: Does it contain resource/mutation patterns?
3. For each match found, add entry to the corresponding map JSON
4. For Map B: scan the file's import statements. Flag any cross-layer violations.
5. Run FM6 detection on any new function bodies (consecutive mutations)

#### If `--dir <path>`:

Same as `--file` but recursively applied to all `.ts`/`.tsx`/`.js`/`.jsx` files in the directory.

#### If `--changed`:

1. Run `git diff --name-only` since the `generatedAt` timestamp in the map files
2. For each changed file, run the `--file` scan
3. For any deleted files, remove their entries from all map JSONs
4. For `package.json` changes: detect added/removed dependencies, run `--pkg` for each

### Step 3: Compute the diff

Compare scanned results against baseline maps:

```
NEW entries:      entries in scan not in baseline → ADD to maps
REMOVED entries:  entries in baseline not in scan → REMOVE from maps (file deleted)
UNCHANGED:        entries present in both → leave as-is
```

### Step 4: Check invariant-derived bans

Do NOT redefine invariants in this skill. `archguard-discover` is the source of truth for the invariant set.

Read the invariant-derived bans from `.archguard/discover.md` and re-evaluate only their
**status** against the changed files/packages:

```
For each invariant ban discovered earlier:
  1. Read its previous status: active / inactive / deferred
  2. Recompute its activation condition against the changed scope
  3. Compare old status vs new status
  4. Report only the delta:
       - activated
       - deactivated
       - unchanged
       - still deferred to clarify
```

Examples of status changes (examples only, NOT hardcoded rules owned by analyze):
- A new outbound HTTP call appears → a timeout-related invariant may become active
- A new runtime/env access appears outside config → an env-boundary invariant may become active
- A new unbounded loop appears → a resilience invariant may become active
- A deleted file removes the last occurrence of a banned primitive → that invariant may become inactive

`archguard-analyze` owns diffing and status updates only. It does **not** define what the invariants are.

### Step 5: Update files

**Merge into each `.archguard/maps/*.json`:**
- `generatedAt` updated
- Entries added, removed, or left unchanged
- Total counts updated

**Append to `.archguard/tech-context.json`:**
- New dependency entries added
- Existing entries left untouched

**Append to `.archguard/discover.md`:**
- Add a `## Analyze Update (<timestamp>)` section at the bottom
- List what was added, removed, activated, deactivated, or left unchanged

### Step 6: Output summary

```
archguard-analyze: <scope> scan complete

  Tech:
    ✓ <package-name>: Q1 triggered → ban: <description> → FM1
    ✓ <package-name>: Q2 triggered → ban: <description> → FM4

  Maps:
    + 3 trust-boundary entries (<file>:<line>)
    + 1 output-surface entry (<file>:<line>)
    + 2 resource-mutation entries (<file>:<line>)
    ~ 0 dependency-graph violations
    - 0 entries removed

  Invariant bans:
    ⚠ <ban name>: <old status> → <new status> (<reason>)
    ✓ <ban name>: unchanged

  Next steps:
    ⚠ 3 new bans added → run archguard-plan to regenerate rules
    ⚠ 1 new trust boundary → run archguard-clarify to confirm FM1 scope
       (or skip if same categories already confirmed)
```

## Output

Analyze modifies these files (merge, never replace):
- `.archguard/maps/trust-boundaries.json`
- `.archguard/maps/dependency-graph.json`
- `.archguard/maps/output-surfaces.json`
- `.archguard/maps/resources-mutations.json`
- `.archguard/tech-context.json`
- `.archguard/discover.md` (appended)

## Constraints

- **Merge, never replace.** Existing map entries are preserved unless their source file was deleted.
- **Report, don't decide.** Analyze tells the user what changed and what to re-run. It does not auto-run clarify or plan.
- **JSON maps are the source of truth.** The markdown discover.md is a human-readable view — the JSON drives the diff logic.
- **Run discover first.** If maps don't exist, abort and tell the user to run archguard-discover.
- **No user questions.** Analyze is read-compute-write only. All decisions are left for clarify.
